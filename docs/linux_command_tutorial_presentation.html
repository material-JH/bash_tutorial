Linux command tutorial

리눅스 터미널을 연구 도구로 쓰기

누리온 로그인 튜토리얼을 먼저 마친 뒤, 같은 서버 화면에서 파일을 찾고, 읽고, 복사하고, 수정하는 입문 발표 자료입니다.

login01에서 pwd → ls → cp → mv

오늘의 목표는 “명령어 암기”가 아니라 누리온 안에서 안전하게 위치 확인, 파일 읽기, 파일 복사/이동을 하는 흐름을 익히는 것입니다.

Roadmap

누리온에 로그인한 상태에서 바로 손으로 확인하기

0:00–0:10

누리온 접속 확인

이미 로그인한 누리온 화면에서 `hostname`, `whoami`, `pwd`를 확인합니다.

0:10–0:25

위치와 파일

`pwd`, `ls`, `cd`, `cp`, `mv`로 현재 위치와 파일 이동을 연습합니다.

0:25–0:38

텍스트 읽기와 `vim`

`more`, `head`, `tail`, `vim`으로 입력 파일과 출력 로그를 확인합니다.

0:38–0:50

안전한 파일 관리

`cp`, `mv`, `rm -i`로 복사, 이름 변경, 삭제를 천천히 연습합니다.

0:50–1:00

출력 읽기와 다음 과제

`grep`, `tail`, `file`로 결과 파일을 읽고 다음 세션 전 homework를 확인합니다.

Mental model

로그인 후 첫 질문: “지금 누리온의 어느 폴더에 있는가?”

local$ 내 컴퓨터

  • 로그인 튜토리얼에서 이미 사용한 터미널
  • 필요하면 다시 SSH를 시작하는 위치
  • 내 Desktop/Finder/Explorer 파일이 있는 곳

server$ 원격 서버

  • 지금 수업을 진행하는 누리온 환경
  • home, scratch, QE 파일이 있는 곳
  • `pwd`, `ls`, `cp`, `mv`, `grep`을 연습하는 곳

중요: 수업 중에는 대부분 server$, 즉 누리온 로그인 노드에서 명령을 입력합니다.

SSH

SSH 로그인은 별도 자료에서 먼저 끝냅니다. 여기서는 접속 확인만 합니다.

# 누리온 로그인 후 확인
server$ hostname
server$ whoami
server$ pwd

# 필요할 때 종료
server$ exit

로그인 후 자주 당황하는 것

  • 프롬프트 모양은 계정/터미널마다 다를 수 있습니다.
  • hostname이 login01처럼 보이면 누리온 로그인 노드입니다.
  • `pwd` 결과가 /home01/... 이면 누리온 home에 있는 것입니다.
Grammar

명령어는 보통 “명령 + 옵션 + 대상”으로 읽습니다.

server$ ls -la examples/qe
#        │  │  └─ 대상: 확인할 폴더
#        │  └──── 옵션: 자세히 + 숨김 파일 포함
#        └─────── 명령: 파일 목록 보기

현재 위치

pwd

지금 내가 어느 폴더에 있는지 확인.

파일 목록

ls -la

권한, 크기, 숨김 파일까지 보기.

경로

~, ., .., /

홈, 현재, 부모, 최상위 위치.

Safe practice

실습은 항상 안전한 폴더와 실제 예제 파일로 시작합니다.

server$ git clone https://github.com/material-JH/bash_tutorial.git ~/bash_tutorial
server$ mkdir -p ~/linux-practice
server$ cp -r ~/bash_tutorial/examples ~/linux-practice/
server$ cd ~/linux-practice/examples
server$ pwd

왜 예제 파일을 미리 넣었나?

  • 초보자는 따옴표와 특수문자 때문에 파일 생성부터 막힐 수 있습니다.
  • 실제 로그, QE 입력/출력, 실행 파일, 바이너리 샘플로 연습합니다.
  • 삭제 연습도 공동 연구 데이터가 아닌 sandbox 안에서만 합니다.
Examples

`examples/` 하나로 대부분의 명령어를 연습합니다.

navigation/
`cd`, `ls`, `tree`, `find`로 폴더 구조를 읽는 연습.
text/
`cat`, `more`, `head`, `tail`, `grep`, `vim`으로 텍스트와 로그 읽기.
files/
`cp`, `mv`, `rm -i`, `rm -r`를 안전하게 연습하는 작은 파일들.
qe/
`si.scf.in`, `si.scf.out`, `si.scf.err`, `fake-pw.x`로 QE 작업 흐름 연습.
binary/
텍스트 파일과 바이너리 파일을 `file`, `hexdump`, `strings`로 구분.
Navigation

서버 작업의 첫 습관: 이동하기 전에 보고, 실행하기 전에 위치 확인

server$ pwd
server$ ls
server$ ls -la
server$ cd examples/qe
server$ cd ..
server$ cd ~
server$ tree .
# tree가 없으면
server$ find . -maxdepth 3 -print

GUI의 주소창을 보는 습관이 터미널에서는 `pwd`입니다.

Text

계산 파일은 먼저 읽고, 필요한 부분만 수정합니다.

cat

짧은 파일

전체가 한 화면에 들어올 때.

more

긴 파일

스페이스로 다음 페이지, `q`로 종료.

head

처음 부분

입력 헤더, 설정 확인.

tail

마지막 부분

계산 종료, 에러 확인.

server$ more text/qe-output-snippet.out
server$ tail -n 40 qe/si.scf.out
Vim

`vim`은 처음에는 “열기, 검색, 저장, 종료”만 익히면 됩니다.

server$ vim qe/si.scf.in

# vim 안에서
/ecutwfc
/K_POINTS
i         # 입력 모드
Esc       # 명령 모드
:wq       # 저장 후 종료
:q!       # 저장하지 않고 종료

초보자에게 가장 중요한 설명

  • 글자가 입력되지 않으면 명령 모드일 수 있습니다.
  • 빠져나오고 싶으면 먼저 Esc를 누릅니다.
  • 수정했는지 확신이 없으면 `:q!`로 나가면 됩니다.
  • QE 입력 파일은 한 글자 오타도 계산 실패로 이어질 수 있습니다.
Files

`rm`은 휴지통이 아닙니다. 삭제 전에는 항상 미리 봅니다.

server$ cd ~/linux-practice/examples/files
server$ cp a.txt a-copy.txt
server$ mv a-copy.txt renamed.txt
server$ ls -l
server$ rm -i renamed.txt

위험 명령 체크리스트

  1. `pwd`로 현재 위치 확인
  2. `ls 대상`으로 삭제될 파일 확인
  3. 처음에는 `rm -i` 사용
  4. `rm -r`, `rm -rf`, `sudo`는 의미를 설명한 뒤 실행
Output flow

터미널의 강점은 “출력을 다음 작업으로 연결”하는 것입니다.

명령 실행

grep "ERROR" app.log

→

파이프로 전달

| wc -l

→

파일에 저장

> error-count.txt

server$ grep "ERROR" text/app.log | wc -l
server$ ./qe/fake-pw.x -in qe/si.scf.in > fake.out 2> fake.err
server$ wc -c fake.err     # 성공하면 보통 0 bytes
server$ ./qe/fake-pw.x -in qe/no-file.in > bad.out 2> bad.err
server$ cat bad.err
server$ ./qe/fake-pw.x -in qe/no-file.in > bad.log 2>&1
Search

계산이 끝났는지는 파일을 열어보는 대신 검색으로 확인합니다.

server$ grep -n "JOB DONE" qe/si.scf.out
server$ grep -n "total energy" qe/si.scf.out
server$ grep -ni "error" qe/si.scf.out
server$ find . -name "*.out"
server$ find . -name "*.out" -exec grep -H "JOB DONE" {} \;

한 줄씩 의미를 붙여서 가르치기

  • `-n`: 줄 번호 표시
  • `-i`: 대소문자 무시
  • `-H`: 여러 파일 검색 시 파일명 표시
  • `{}`: `find`가 찾은 파일 이름이 들어가는 자리
Executables

터미널 명령도 결국 서버 어딘가에 있는 실행 파일입니다.

server$ command -v python3
server$ command -v vim
server$ type python3
server$ file scripts/hello.sh
server$ file qe/fake-pw.x

출력이 없으면?

  • 명령어가 설치되지 않았거나 현재 환경에서 찾을 수 없습니다.
  • HPC 서버에서는 `module avail quantum`을 먼저 확인할 수 있습니다.
  • 서버마다 사용 가능한 명령어와 설치 위치가 다를 수 있습니다.
Permissions

실행 권한은 파일이 “프로그램처럼 실행될 수 있음”을 뜻합니다.

server$ ls -l scripts/hello.sh
server$ ./scripts/hello.sh
server$ chmod +x scripts/hello.sh
server$ ./scripts/hello.sh

`sudo`는 입문 실습 명령이 아닙니다.

공용 서버에서 관리자 권한 명령을 임의로 실행하지 않습니다. 설치나 권한 문제가 있으면 서버 관리자 또는 튜토리얼 담당자에게 정확한 에러 메시지를 전달합니다.

Scientific files

계산 과학 파일도 대부분 “읽을 수 있는 텍스트”에서 시작합니다.

입력 파일

*.in
parameters
structure

→

출력 파일

*.out
*.err
log text

→

읽고 검색

tail
grep
more

오늘은 계산 제출이 아니라, 입력/출력 파일을 안전하게 읽고 찾는 방법까지만 배웁니다.

Outputs

출력 파일에서는 필요한 줄을 검색해서 찾습니다.

server$ tail -n 40 si.scf.out
server$ grep -n "error" si.scf.out
server$ grep -n "warning" si.scf.out
server$ grep -n "total energy" si.scf.out
server$ more si.scf.out

텍스트와 바이너리를 구분

  • `.in`, `.out`, `.err`는 보통 텍스트입니다.
  • wavefunction, charge density, `.dat` 일부는 바이너리일 수 있습니다.
  • 먼저 `file filename`으로 종류를 확인합니다.
  • 바이너리를 `vim`으로 열어 수정하지 않습니다.
Troubleshooting

실패했을 때는 추측하지 말고 체크리스트를 따라갑니다.

1. 위치

`pwd`, `ls -la`로 입력 파일이 있는 폴더인지 확인합니다.

2. 파일 이름

`ls`로 파일 이름과 확장자를 정확히 확인합니다.

3. 입력 파일

`vim filename`으로 필요한 줄을 천천히 확인합니다.

4. stderr

`cat si.scf.err`, `tail -n 80 si.scf.out`을 먼저 읽습니다.

5. 검색

`grep -ni "error"`처럼 필요한 단어를 검색합니다.

6. 질문

질문할 때는 명령어, 현재 폴더, 입력 파일, 마지막 80줄 로그를 함께 보냅니다.

Appendix 1

QE를 빨리 쓰려면 패키지나 서버 모듈을 먼저 확인합니다.

Conda / Mamba

server$ conda create -n qe -c conda-forge qe
server$ conda activate qe
server$ pw.x -h

conda-forge 패키지 이름은 `qe`입니다.

Ubuntu / Debian apt

server$ sudo apt update
server$ sudo apt install quantum-espresso quantum-espresso-data
server$ pw.x -h

공용 서버에서는 임의로 `sudo`를 쓰지 않습니다.

HPC 서버 모듈

server$ module avail quantum
server$ module load quantum-espresso
server$ command -v pw.x

연구실 서버에서는 보통 이 방법이 가장 안전합니다.

패키지 설치는 편하지만 버전이나 성능이 제한될 수 있습니다. 그래서 큰 계산용 서버에서는 컴파일 버전을 쓰기도 합니다.

Appendix 2

Intel 컴파일은 “컴파일러 환경 → configure → make → 검증” 순서입니다.

# Intel oneAPI + MPI wrapper 예시
server$ source /opt/intel/oneapi/setvars.sh
server$ git clone https://gitlab.com/QEF/q-e.git
server$ cd q-e
server$ ./configure --enable-parallel CC=mpiicx F90=mpiifx MPIF90=mpiifx
server$ make pw -j 8
server$ ./bin/pw.x -h

# 예전 Intel 환경이면 mpiicc / mpiifort 이름일 수 있음

주의사항

  • 새 Intel oneAPI는 `icx/ifx`, MPI wrapper는 `mpiicx/mpiifx`를 주로 씁니다.
  • 예전 Intel Parallel Studio 환경은 `icc/ifort`, `mpiicc/mpiifort`일 수 있습니다.
  • BLAS/LAPACK/FFT/MKL/MPI 설정은 서버마다 다르므로 `configure` 출력과 `make.inc`를 확인합니다.
  • 컴파일 후에는 `command -v pw.x`, `pw.x -h`, 짧은 SCF 예제로 실제 실행을 확인합니다.
